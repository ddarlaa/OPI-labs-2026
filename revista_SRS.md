## 1. Введение
**Проект:** Reverso.net  
**Версия документа:** 1.0

Целью данного документа является описание требований к веб-приложению Reverso.net. Система представляет собой набор лингвистических инструментов: переводчик, контекстный словарь, проверка орфографии.

## 2. Общее описание
**Акторы (Роли):**
* **Гость:** Незарегистрированный пользователь.
* **Зарегистрированный пользователь:** Пользователь с аккаунтом.
* **Premium-пользователь:** Платный подписчик.
* **Администратор:** Управление системой.

## 3. Спецификация требований

### 3.1. Требования пользователей (User Requirements - UR)

| ID        | Описание требования                                               | Приоритет | Оценка (ч) |
|:----------|:------------------------------------------------------------------|:----------|:----------:|
| **UR-01** | Пользователь хочет переводить текст с одного языка на другой.     | High      |     -      |
| **UR-02** | Пользователь хочет видеть примеры использования слов в контексте. | High      |     -      |
| **UR-03** | Пользователь хочет проверять текст на ошибки.                     | Medium    |     -      |

### 3.2. Требования владельцев (Business Rules - BR)

| ID        | Описание требования                                               | Приоритет | Оценка (ч) | Аргументация                       |
|:----------|:------------------------------------------------------------------|:----------|:----------:|:-----------------------------------|
| **BR-01** | Монетизация через баннерную рекламу для бесплатных пользователей. | High      |     20     | Интеграция AdSense/RTB блоков.     |
| **BR-02** | Конвертация пользователей в Premium подписку.                     | High      |     40     | Разработка лендинга и ограничений. |

### 3.3. Функциональные требования (Functional Requirements - FR)

| ID        | Описание требования                                                                         | Атрибуты RUP                               | Оценка (ч) | Аргументация                                    |

### 3.4. Нефункциональные требования (Non-functional Requirements - NFR)

| ID | Описание требования | Приоритет | Оценка (ч) |
|:---|:---|:---|:---:|
| **NFR-01** | **Скорость.** Отклик перевода < 1 сек. | High | 40 |

## 4. Диаграммы прецедентов (Use Case)

Ниже представлена диаграмма, описывающая основные сценарии использования.

```mermaid
usecaseDiagram
    actor "Гость" as Guest
    actor "Пользователь" as RegUser
    actor "API Перевода" as API

    package "Веб-сайт Reverso" {
        usecase "Перевести текст" as UC1
        usecase "Определить язык" as UC1_1
        usecase "Найти в контексте" as UC2
        usecase "Проверить орфографию" as UC3
        usecase "Вход / Регистрация" as UC4
        usecase "Сохранить в словарь" as UC5
        usecase "Купить Premium" as UC6
    }

    Guest <|-- RegUser

    Guest --> UC1
    Guest --> UC2
    Guest --> UC3
    Guest --> UC4

    UC1 ..> UC1_1 : <<include>>
    UC1 --> API : запрос

    RegUser --> UC5
    RegUser --> UC6
```

### Описание диаграммы
Пользователь может выполнять перевод текста (включает определение языка), искать контекст и проверять орфографию. Зарегистрированный пользователь наследует права гостя и дополнительно может сохранять слова и покупать подписку.

## 5. Выводы
В ходе работы составлен документ SRS согласно методологии RUP. Требования разделены на функциональные, пользовательские и нефункциональные. Общая оценка трудозатрат на реализацию MVP составляет порядка **800-1000 часов**, что требует команды из 3-4 разработчиков на срок около 2 месяцев.
